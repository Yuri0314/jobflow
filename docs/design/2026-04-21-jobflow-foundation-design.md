# jobflow 基础设计

日期：2026-04-21  
状态：已批准，用于初始化仓库与后续细化设计

## 1. 产品定义

`jobflow` 是一个面向个人求职场景的 CLI capability runtime。

它的目标不是提供一个完整的自有 agent 产品形态，也不是优先发展成对话式终端 agent。  
它的核心目标是提供一组稳定、可组合、可被外部 agent 调用的求职领域能力。

系统要帮助用户完成这些事情：

- 接收来自招聘网站和其他来源的机会信息
- 将机会信息标准化并保存为一致的本地数据模型
- 根据个人目标、偏好和约束对机会进行评分
- 跟踪求职流程状态和下一步动作
- 通过稳定命令与结构化输出，为成熟 agent 系统提供可调用能力

这个系统不被定义为“自动投递产品”。  
它被定义为“个人求职工作流自动化 runtime”。

## 2. 第一阶段范围

第一阶段包含：

- CLI runtime
- 作为轻量采集与触发层的浏览器插件
- 求职实体的共享 schema
- CLI 与插件之间通信的共享 protocol
- 面向外部 agent 调用的稳定命令接口与结构化输出约定
- 用于说明系统边界和演进路径的设计文档

第一阶段不包含：

- 独立的 Web 前后端应用
- 平台相关的自动投递
- 深度反爬自动化和 stealth 执行
- 从零开始构建一个完整通用型 agent shell
- 以自然语言对话体验为主的本地 agent 宿主

## 3. 设计原则

### 3.1 以用户为中心，而不是以平台为中心

系统服务的是求职者，而不是某一个招聘平台。  
它应维护中立的、跨来源的状态与决策支持，而不是围绕单个平台的站内流程优化。

### 3.2 核心稳定，输入可替换

CLI runtime 是长期稳定的核心。  
招聘网站是高价值输入源，但系统不能依赖某一个平台才能保持有用。

### 3.3 浏览器层保持轻量

浏览器插件应尽量轻。  
它的主要职责是采集页面上下文、触发接入动作，并提供少量现场反馈。

### 3.4 优先做能力层，而不是自建 agent 壳子

`jobflow` 的长期价值在于一组稳定的求职领域能力，而不是一个新的通用 agent 交互壳子。  
第一阶段优先保证：

- 能力边界清楚
- 命令稳定且可组合
- 输出结构化且便于 agent 调用

是否提供本地 agent mode 不是当前主目标。

### 3.5 对 skill / tool 集成友好

`jobflow` 应优先适配成熟 agent 系统的接入方式。  
因此第一阶段要默认考虑：

- 可预测的命令语义
- 结构化 JSON 输出
- 清晰的错误信息
- 稳定的 schema 与 protocol

## 4. 顶层架构

仓库采用单仓多模块结构：

```text
jobflow/
  apps/
    cli/
    browser-extension/
  packages/
    schema/
    protocol/
  docs/
    design/
  scripts/
```

### 4.1 `apps/cli`

负责：

- CLI 入口
- 本地 runtime 行为
- 本地状态存储
- `jobflow` 核心命令
- 求职能力实现
- 面向外部 agent 的稳定调用面

### 4.2 `apps/browser-extension`

负责：

- 招聘网站页面上下文采集
- 由用户显式触发的数据接入动作
- 页面级别的轻量状态展示和轻交互
- 基于共享 protocol 与 CLI runtime 通信

### 4.3 `packages/schema`

负责共享领域模型，例如：

- 职位接入 payload
- 标准化后的职位记录
- pipeline 条目
- 简历引用
- 评分结果

### 4.4 `packages/protocol`

负责浏览器插件、CLI 与未来外部接入之间的共享通信契约，包括：

- 消息封装格式
- 版本标记
- 命令请求/响应结构
- 能力名与对应 payload schema

## 5. 核心能力模型

第一阶段核心能力包括：

- `ingest_job`  
  接收原始机会输入，来源可以是链接采集、页面提取或粘贴的 JD 文本。

- `normalize_job`  
  把原始输入转换成稳定的结构化记录。

- `score_job`  
  按用户目标、约束和偏好对机会进行评估。

- `track_pipeline`  
  持久化状态、下一步动作、跟进日期和备注。

- `summarize_next`  
  生成当前最重要下一步动作的简洁视图。

- `resume_context`  
  存储并引用简历版本和目标偏好，但第一阶段不实现完整自动简历定制。

## 6. 浏览器插件职责

浏览器插件不应成为主要智能层。

它应该做的事情：

- 识别相关职位详情页或沟通页
- 提取当前页面可见的结构化上下文
- 允许用户触发保存或同步动作
- 把采集结果发送给 CLI runtime
- 展示轻量状态，例如“已保存”“已评分”“待跟进”

它不应做的事情：

- 复制整套评分和工作流引擎
- 维护长期记忆
- 实现深度自主浏览或反爬逻辑

## 7. CLI Runtime 职责

CLI runtime 是系统核心。

它应该做的事情：

- 维护本地持久化状态
- 暴露稳定的命令和能力接口
- 执行标准化和评分逻辑
- 管理求职流程状态迁移
- 提供摘要和下一步视图
- 同时支持人类直接使用和外部 agent 间接调用

当前阶段，CLI 的首要目标不是长成一个完整的对话式 agent 宿主。  
它的首要目标是成为一个稳定的求职能力层。

## 8. 数据流

第一阶段主流程：

1. 用户正在查看一个职位页面
2. 浏览器插件采集页面上下文
3. 插件根据共享 protocol 发送 ingest payload
4. CLI runtime 存储原始输入、执行标准化，并计算评分
5. CLI runtime 记录或更新 pipeline 状态
6. 用户或外部 agent 通过 CLI 查询下一步动作

次级流程包括：

- 直接把 JD 文本粘贴到 CLI
- 直接把链接导入 CLI
- 由外部 agent 调用 CLI 能力

## 9. 命令面方向

初始命令面应优先选择稳定、可组合、便于外部调用的命令，而不是一开始就做一个大型交互式 shell。

例如：

- `jobflow ingest`
- `jobflow normalize`
- `jobflow score`
- `jobflow pipeline`
- `jobflow next`

命令设计要满足这些要求：

- 支持结构化输入
- 支持结构化 JSON 输出
- 输出字段稳定，便于 skill/tool 集成
- 错误信息可预测，便于 agent 处理

## 10. 第一阶段非目标

第一阶段应明确避免这些方向：

- 重建完整 Web 产品壳子
- 把系统价值紧耦合到某一个招聘网站
- 过早重投入到自主平台自动化
- 自己重做一个通用 agent 壳子
- 用 agent 语言掩盖不清晰的领域边界

## 11. 演进路径

计划中的演进顺序：

1. 先确立 CLI runtime 的边界、能力契约和结构化输出约定
2. 再增加浏览器插件的采集工作流
3. 让 CLI 更容易被成熟 agent 系统通过 tool / skill 方式调用
4. 在外部集成稳定后，再决定是否需要补充更轻量的本地 agent mode
5. 最后才评估是否真的需要独立 UI 产品

## 12. 待定决策

这些问题在基础设计之后有意延后，不在当前文档中定死：

- 本地存储的具体实现方式
- CLI 与插件分别使用什么语言 / runtime
- 插件与 CLI 之间具体采用什么进程间通信机制
- 第一版评分策略做到多深
- 外部 skill 集成优先适配哪些 agent 系统
