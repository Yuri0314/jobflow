# jobflow 桌面端内嵌浏览器方向设计

日期：2026-05-07  
状态：未来路线设计草案，不进入当前实现阶段

## 1. 文档目的

本文档用于记录一个后续可能方向：在 `jobflow` CLI runtime 稳定之后，增加一个带内嵌浏览器的桌面端入口。

这个方向不是要推翻当前 CLI-first 架构，而是为 `jobflow` 增加一个更强的本地使用外壳。

## 2. 基本判断

桌面端内嵌浏览器是合理方向，但不应成为当前第一阶段主线。

当前 `jobflow` 的核心价值仍然是：

- 本地求职数据模型
- 职位接入与标准化
- 评分
- pipeline 状态
- 简历引用
- 稳定 CLI / protocol 能力

桌面端应该依附于这些能力，而不是替代它们。

## 3. 推荐定位

未来桌面端可以被定义为：

```text
jobflow 的强入口 shell
```

它负责：

- 提供内嵌浏览器
- 帮用户浏览招聘网站
- 采集当前页面上下文
- 展示 `jobflow` 已知状态
- 触发保存、评分、pipeline 更新等能力

它不应该在初期负责：

- 维护独立数据模型
- 复制 CLI runtime 逻辑
- 直接变成自动批量投递工具
- 绕过招聘平台风控
- 自己实现一套 agent shell

## 4. 与浏览器插件的关系

浏览器插件和桌面端不是互斥关系。

可以这样区分：

```text
浏览器插件：轻入口
桌面端：强入口
CLI runtime：核心能力层
agent / tool：外部智能调用层
```

浏览器插件适合：

- 用户已经在常用浏览器里看职位
- 轻量保存当前页面
- 少量状态提示

桌面端适合：

- 想把求职流程集中到一个本地工作台
- 希望页面浏览、职位状态、评分、下一步动作放在一起
- 后续可能需要更强的自动化能力边界

## 5. 分级演进

### Level 1：内嵌浏览器 + 手动采集

用户在桌面端打开招聘网站，看到职位后手动点击保存。

桌面端采集：

- 当前 URL
- 页面标题
- 公司名称
- 页面可见文本
- 来源站点

然后调用现有能力：

```text
jobflow ingest
jobflow normalize
jobflow score
```

这是最安全、最符合当前架构的第一步。

### Level 2：半自动工作台

桌面端可以把职位列表和 `jobflow` 状态结合起来：

- 显示是否已保存
- 显示评分
- 显示 pipeline 状态
- 显示下一步动作
- 支持人工确认后的批量保存或忽略

关键动作仍由用户确认。

这个阶段的目标是“提高判断和整理效率”，不是自动代替用户投递。

### Level 3：自动化操作

桌面端可能进一步支持：

- 自动翻页
- 自动筛选
- 自动点击部分流程
- 自动打招呼或投递

这个阶段风险最高。

进入这个阶段前必须重新评估：

- 招聘平台服务条款
- 账号风控风险
- 用户确认机制
- 操作频率限制
- 失败恢复机制
- 是否会把产品定位带偏

第一阶段和第二阶段不应依赖 Level 3。

## 6. 技术选型方向

候选方案：

- Electron
- Tauri
- 其他本地 WebView shell

当前不做最终选型。

如果优先考虑和现有 TypeScript 代码复用，Electron 上手成本更低。  
如果优先考虑体积、性能和系统集成，Tauri 值得后续评估。

不管选哪个，核心原则是：

- 桌面端调用 `packages/schema` 和 `packages/protocol`
- 桌面端不复制核心业务逻辑
- 长期状态仍以 `jobflow` runtime 为准
- 页面采集层和求职数据层分离

## 7. 与 GetJob 类产品的差异

类似 GetJob 的产品更接近：

```text
桌面软件 + 浏览器自动化 + 平台投递
```

`jobflow` 更应该保持：

```text
本地 runtime + 求职状态系统 + 可组合能力层
```

如果未来做桌面端，也不应一开始就以“自动批量投递”为核心卖点。

更稳妥的方向是：

1. 先做采集
2. 再做状态和决策辅助
3. 最后再评估自动化操作

## 8. 当前阶段建议

当前不实现桌面端。

建议先完成：

- CLI runtime 稳定
- resume 命令补齐
- 本地状态 inspect / export
- browser extension 或协议对接脚本

等这些能力稳定后，再创建 `apps/desktop`。

## 9. 未来最小可验证桌面端

如果后续启动桌面端，最小验证版本可以只做：

1. 打开一个内嵌浏览器窗口
2. 用户输入或打开招聘页面 URL
3. 点击“保存当前职位”
4. 桌面端提取页面 URL 和可见文本
5. 调用 `jobflow ingest`
6. 显示返回的 `ingest_id`

只要这条链路成立，桌面端方向就具备继续验证的价值。
